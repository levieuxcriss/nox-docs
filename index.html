Get up and running with nox dev scripts in minutes

  • FiveM server with a supported framework (ESX, QBCore, QBox, VORPCore, or ND_Core)

  • Discord webhook URL (optional, for notifications)

⚙️ Framework Support

Automatic detection — no configuration required

All nox dev scripts automatically detect your framework on startup. The detection checks for specific resources in this order:

Detection Order

PriorityFrameworkDetection MethodTables Wiped
1QBCore / QBoxGetResourceState('qb-core')players, player_*, bank_accounts, apartments, crypto
2ESXGetResourceState('es_extended')users, owned_vehicles, billing, society_*, datastore_data
3VORPCoreGetResourceState('vorp_core')characters, horses, inventories, vorp_*
4ND_CoreGetResourceState('ND_Core')nd_characters, nd_vehicles, nd_inventory

How It Works

On server startup, the script checks for each framework in order. When a match is found:

  1. The framework name is logged to the console
  2. Framework-specific table lists are loaded for character wipes
  3. Appropriate identifier extraction methods are configured
Note: If no framework is detected, the script will log a warning but continue running. Character wipes will not function without a detected framework.

Adding a New Language

locales/xx.lua (your new language)
Locales['xx'] = {
    crash_detected = "Your translated text here",
    player_died = "Player death message",
    player_survived = "Survival message",
    -- Copy all keys from en.lua and translate them
}

How the L() Function Works

The L(key) function is used throughout the scripts to fetch translated strings:

  1. Checks for the key in the current locale (e.g., Locales['fr'])
  2. Falls back to English (Locales['en']) if key not found
  3. Returns the key name itself as last resort
FxAP Compatible: All locale files are in the escrow_ignore list, so they remain editable even after FxAP protection.

🔒 FxAP Protection

Protect your code while keeping configs editable

What is FxAP?

FiveM Asset Protection (FxAP) encrypts your script's core logic (client.lua, server.lua) while allowing specific files to remain editable by the end user.

Protected vs. Editable

FileStatusDescription
client.lua🔒 EncryptedClient-side logic
server.lua🔒 EncryptedServer-side logic, wipe functions
config.lua🔓 EditableAll configuration options
locales/*.lua🔓 EditableAll 20 language files

fxmanifest.lua Setup

fxmanifest.lua
fx_version 'cerulean'
game 'gta5'

escrow_ignore {
    'config.lua',
    'locales/*.lua'
}
For Developers: When you submit your script to FxAP, the files listed in escrow_ignore will remain unencrypted. Everything else gets encrypted automatically.

💀 fatal_crash — Overview

High-speed crash detection with character permadeath

v4.0

What It Does

fatal_crash monitors player vehicle speed. When a player crashes at high speed (default: 120 km/h), there's a 50/50 chance to:

  • Survive: Player receives a notification that they survived
  • Die + Wipe: Player dies AND their entire character is permanently deleted from the database
⚠️ EXTREME: This is a hardcore permadeath script. The character wipe is permanent and irreversible. The player must create a completely new character.

How The Wipe Works

  1. Crash Detection: Client detects sudden speed drop above threshold
  2. 50/50 Roll: Server generates random outcome
  3. Kill Player: If unlucky, player health is set to 0
  4. Database Wipe: All character data deleted from framework-specific tables
  5. Mark Player: Player is recorded in fatal_crash_wiped tracking table
  6. Reconnect Protection: On next join, deferrals intercept and re-wipe any data the framework recreated

Features

🚗

Speed Detection

Configurable speed threshold with accurate crash detection algorithm

🎲

50/50 Chance

Fair random roll — survive or lose everything

🔄

Mark & Re-wipe

Prevents frameworks from restoring wiped data on reconnect

💬

Discord Alerts

Rich embed notifications for all crash events

☠️ death_wipe — Overview

Death counter with permanent character wipe

v4.0

What It Does

death_wipe tracks every player death. After reaching a configurable number of deaths (default: 5), the player's character is permanently wiped from the database.

How It Works

  1. Player Dies: Death count increments by 1
  2. Counter Display: Player sees "Death X/5" notification
  3. Threshold Reached: When deaths reach the limit, character is fully wiped
  4. Mark & Track: Player stored in death_wipe_marked table
  5. Reconnect Protection: Framework-recreated data is re-wiped on next join

Features

🔢

Death Counter

Persistent counter stored in database, survives restarts

⚙️

Configurable Limit

Set any number of deaths before wipe (default: 5)

👮

Admin Commands

Check, set, and reset death counts for any player

💬

Discord Alerts

Notifications for every death and wipe event

Usage Examples

In-game commands
-- Check player ID 3's death count
/deathcount 3

-- Set player ID 3 to 2 deaths
/setdeaths 3 2

-- Reset player ID 3 back to 0
/resetdeaths 3
Permissions: Admin commands use your framework's built-in permission system (ACE permissions for ESX, admin groups for QBCore, etc.)
